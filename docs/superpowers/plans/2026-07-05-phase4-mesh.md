# Aeon-V2 Phase 4: Agent Mesh Integration Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Aeon-V2 becomes a first-class Agent Mesh peer — other agents (claude@x1, claude@t3610) can address it and get AgentLoop-powered replies — and the model router can borrow LLM workers from other mesh machines.

**Architecture:** New `aeon.mesh` package. `MeshClient` is a stdlib HTTP wrapper over the existing hub protocol (`POST /api/heartbeat`, `GET /api/inbox/{agent}?after=`, `POST /api/messages`, `POST /api/messages/{id}/ack`). `MeshPeer` polls its inbox and, for each addressed message, runs `AgentLoop` and posts the collected reply — replacing the `scripts/aeon_bridge.py` PTY bridge with a native loop. A `mesh_message` tool lets Aeon post to threads during a conversation. Worker discovery probes candidate OpenAI-compatible endpoints on other machines and registers the healthy ones with `ModelRouter`.

**Tech Stack:** unchanged (stdlib + FastAPI).

## Global Constraints

- Hub auth: `Authorization: Bearer <AEON_MESH_TOKEN>`.
- Env: `AEON_MESH_HUB` (e.g. `http://127.0.0.1:8787`), `AEON_MESH_TOKEN`, `AEON_MESH_AGENT_ID` (default `aeon@<hostname>`), `AEON_MESH_MACHINE` (default hostname), `AEON_MESH_LLM_WORKERS` (colon-separated base URLs).
- The mesh peer only ever replies with AgentLoop text output; it does not auto-approve gated tools (approvals still surface through the normal broker/UI).
- All HTTP isolated behind an injectable `http_request` callable for tests — no real sockets in the suite.
- Commit after every task; suite green.

### Task 1: `aeon.mesh.client` — hub HTTP wrapper

**Files:** `server/src/aeon/mesh/__init__.py`, `server/src/aeon/mesh/client.py`, test `server/tests/test_mesh_client.py`

**Interfaces:**
- `MeshConfig` dataclass built from env: `hub, token, agent_id, machine`. `MeshConfig.from_env() -> MeshConfig`; `agent_id`/`machine` default to `aeon@<socket.gethostname()>` / hostname.
- `class MeshClient(mesh_config, http_request=None)` where `http_request(method, url, headers, body) -> object` defaults to a urllib impl.
  - `.configured -> bool` (hub and token both set)
  - `.heartbeat(status: str) -> None`
  - `.inbox(after: int = 0) -> list[dict]` → `GET /api/inbox/{agent}?after=`
  - `.post_message(thread_id, recipient, content, kind="reply") -> dict`
  - `.ack(message_id) -> None`
- Tests: a fake `http_request` records calls; assert URL/method/body/headers for each method; `configured` false when env missing.

### Task 2: `mesh_message` tool

**Files:** `server/src/aeon/tools/mesh.py`, register in `aeon/tools/__init__.py`, extend `server/tests/test_v2_tools.py`

- `mesh_post(arguments, config)` — `{thread_id, recipient, content}` → posts via `MeshClient(MeshConfig.from_env())`; returns `{"posted": True, "message_id": ...}` or `{"error": "mesh not configured"}` when unconfigured. approval_required=False (posting a chat message is low-risk and mesh is tailnet-only).
- Tests: unconfigured → error dict; configured (monkeypatched MeshClient) → posts and returns id.

### Task 3: `aeon.mesh.peer` — native peer loop

**Files:** `server/src/aeon/mesh/peer.py`, test `server/tests/test_mesh_peer.py`

**Interfaces:**
- `class MeshPeer(config, mesh_client, loop=None, poll_seconds=1.0)` (loop = `AgentLoop`; created if omitted).
- `.poll_once() -> int` — heartbeat idle, fetch inbox after cursor; for each message: heartbeat running, run `AgentLoop.run([{role:user, content: message['content']}])`, join text events into a reply, `post_message(thread_id=msg['thread_id'], recipient=msg['sender'], content=reply)`, `ack(msg['id'])`, advance cursor to max id. Returns count processed. Exceptions per-message are caught and posted back as an error reply so one bad message can't wedge the loop.
- `.run_forever(should_stop=lambda: False)` — poll/sleep loop.
- Tests with a fake MeshClient (canned inbox then empty) and a stub loop: reply posted to the right thread/recipient, message acked, cursor advances, second poll returns 0; a message whose loop raises still gets an error reply + ack.

### Task 4: worker discovery → router

**Files:** modify `server/src/aeon/models/router.py`, test extends `server/tests/test_models_router.py`

- `discover_workers(candidates: list[str], http_probe=None) -> list[Worker]` — for each base URL, `GET {url}/models`; on success build `Worker(name=host, base_url=url, models=["*"], priority=5)`; skip failures. `http_probe(url)` injectable.
- `ModelRouter.add_workers(workers)` — append, dedupe by base_url.
- `ModelRouter` reads `AEON_MESH_LLM_WORKERS` at load and runs discovery (best-effort, probe failures ignored) so mesh GPUs join automatically.
- Tests: two candidates, one healthy → one worker added; env-driven discovery with injected probe; dedupe.

### Task 5: mesh API + console script

**Files:** modify `server/src/aeon/api/app.py`, `server/pyproject.toml` (script `aeon-mesh-peer = "aeon.mesh.peer:main"`), `server/src/aeon/mesh/peer.py` (`main()`), test extends `server/tests/test_api.py`

- `GET /api/mesh` → `{agent_id, machine, configured, workers: [...]}`.
- `POST /api/mesh/message` `{thread_id, recipient, content}` → posts through app-state MeshClient; 503 if unconfigured.
- `main()` builds config, MeshClient, MeshPeer and runs `run_forever()` (the systemd service in Phase 6 runs this).
- Tests: `/api/mesh` shape; message endpoint with a fake client; 503 when unconfigured.
